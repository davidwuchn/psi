{:steps [{:name "prep"
          :workflow "builder"
          :session {:input {:from :workflow-input}
                    :reference {:from :workflow-original}
                    :skills ["work-independently"]}
          :prompt "Prepare the selected PR branch worktree described by $INPUT. Work independently.\n\nRequired procedure:\n1. Read the upstream handoff to identify the PR number, PR URL, branch, base branch, and desired worktree description.\n2. Create or reuse a branch-specific worktree for the PR branch. Reuse in place if an appropriate worktree already exists for that branch.\n3. In that worktree, fetch `origin` and ensure the PR branch is checked out locally.\n4. Confirm the local branch tracks the expected PR branch and report the current HEAD commit.\n5. Do not rebase automatically unless it is required just to make the branch operable; this workflow is about healing checks, not routine branch rebasing.\n6. If prep fails at any point, stop and report the failure instead of continuing on a stale or ambiguous branch state.\n\nOutput requirements:\n- Output a compact Markdown handoff with these headings exactly:\n  - `## Prep Outcome`\n  - `## Worktree State`\n  - `## Handoff Data`\n- Under `## Handoff Data`, include machine-friendly bullet lines for:\n  - `pr_number:`\n  - `pr_url:`\n  - `pr_branch:`\n  - `pr_base_branch:`\n  - `worktree_path:`\n  - `head_commit:`\n  - `prep_status:`"}
         {:name "inspect"
          :workflow "builder"
          :session {:input {:from {:step "prep" :kind :accepted-result}}
                    :reference {:from :workflow-original}
                    :skills ["work-independently"]}
          :prompt "Inspect the current GitHub check state for the prepared PR described by $INPUT. Work independently.\n\nGoal: classify the PR as exactly one of PASS, PENDING, FAIL, or BLOCKED. Success for this workflow means all checks pass; pending checks are not success.\n\nRequired procedure:\n1. Read the upstream handoff to identify the PR number, PR URL, branch, worktree path, and current HEAD commit.\n2. Run `gh pr checks <pr> --json name,state,bucket,link,workflow` and inspect the full current check set.\n3. Also inspect PR metadata as needed with `gh pr view <pr> --json statusCheckRollup,headRefOid,headRefName,url`.\n4. Make run-id discovery explicit rather than implicit:\n   - use the PR branch from the handoff as authoritative\n   - list recent Actions runs for that branch with `gh run list --branch <pr-branch>`\n   - correlate failing or rerun-candidate checks with recent runs by workflow name, recency, and any linked check URLs\n   - when a likely run is found, inspect it with `gh run view <run-id> --json jobs` to confirm job names match the failing check context\n   - if correlation remains ambiguous, say so explicitly instead of inventing run ids\n5. Classify as:\n   - `PASS` when all checks are in passing or equivalent successful/skipped terminal states and none are pending or failing.\n   - `PENDING` when any checks are still queued, in progress, or otherwise non-terminal.\n   - `FAIL` when any checks are failing, cancelled, or otherwise terminally unsuccessful.\n   - `BLOCKED` when the check state cannot be reliably inspected or the PR/worktree state is too inconsistent to proceed safely.\n6. Summarize failing checks and pending checks separately.\n7. For each failing check, include the strongest immediately available clue about whether it looks like a real code/test failure, a flaky test, or infrastructure noise.\n8. Include candidate run ids only when the check-to-run mapping is reasonably supported by branch, workflow, job-name, and recency evidence.\n\nOutput requirements:\n- Output a compact Markdown summary with these headings exactly:\n  - `## Check Outcome`\n  - `## Failing Checks`\n  - `## Pending Checks`\n  - `## Handoff Data`\n- Under `## Handoff Data`, include machine-friendly bullet lines for:\n  - `pr_number:`\n  - `pr_url:`\n  - `pr_branch:`\n  - `worktree_path:`\n  - `head_commit:`\n  - `check_overall_status:`\n  - `failing_checks:`\n  - `pending_checks:`\n  - `rerun_candidate_checks:`\n  - `candidate_run_ids:`\n\nSet `check_overall_status:` to exactly one of `PASS`, `PENDING`, `FAIL`, or `BLOCKED`. Use terse list-like text for `failing_checks`, `pending_checks`, `rerun_candidate_checks`, and `candidate_run_ids`."
          :judge {:system-prompt "You are a workflow routing judge. Respond with exactly one word: PASS, PENDING, FAIL, or BLOCKED."
                  :prompt "Respond exactly with one word: PASS, PENDING, FAIL, or BLOCKED.\n\nRead the actor output and route strictly from the `check_overall_status` in its `## Handoff Data`.\n- Return PASS only when all checks are passing and none are pending or failing.\n- Return PENDING when any checks are still running or queued.\n- Return FAIL when any checks are failing, cancelled, or otherwise terminally unsuccessful.\n- Return BLOCKED when the actor output says the PR/check state could not be inspected reliably or could not be proceeded with safely.\n\nDo not infer a different classification from narrative prose when the handoff data is explicit."}
          :on {"PASS"    {:goto :done}
               "PENDING" {:goto "wait" :max-iterations 24}
               "FAIL"    {:goto "repair" :max-iterations 8}
               "BLOCKED" {:goto :done}}}
         {:name "wait"
          :workflow "builder"
          :session {:input {:from {:step "inspect" :kind :accepted-result}}
                    :reference {:from :workflow-original}
                    :skills ["work-independently"]}
          :prompt "Wait for the selected PR's GitHub checks to reach a more informative state. Work independently.\n\nRequired procedure:\n1. Read the upstream handoff to identify the PR number, PR URL, branch, worktree path, and current check classification.\n2. Use `gh pr checks <pr> --watch --interval 10` as the canonical waiting mechanism. Let it block until checks complete or transition to a terminal state.\n3. After watch completes, immediately capture a fresh structured snapshot with `gh pr checks <pr> --json name,state,bucket,link,workflow`.\n4. Classify the post-watch state as PASS, PENDING, FAIL, or BLOCKED using the same meanings as the inspect step.\n5. Summarize whether the post-wait state now appears passing, failing, still pending, or operationally unclear.\n6. Do not modify code, rerun checks, or push from this step. This step only waits and reports.\n\nOutput requirements:\n- Output a compact Markdown summary with these headings exactly:\n  - `## Wait Outcome`\n  - `## Post-Wait Checks`\n  - `## Handoff Data`\n- Under `## Handoff Data`, include machine-friendly bullet lines for:\n  - `pr_number:`\n  - `pr_url:`\n  - `pr_branch:`\n  - `worktree_path:`\n  - `post_wait_snapshot:`\n  - `watch_status:`\n  - `check_overall_status:`\n\nKeep `post_wait_snapshot` terse but sufficient for routing. Set `check_overall_status:` to exactly one of `PASS`, `PENDING`, `FAIL`, or `BLOCKED`."
          :judge {:system-prompt "You are a workflow routing judge. Respond with exactly one word: PASS, PENDING, FAIL, or BLOCKED."
                  :prompt "Respond exactly with one word: PASS, PENDING, FAIL, or BLOCKED.\n\nRead the actor output and route strictly from the `check_overall_status` in its `## Handoff Data`.\n- Return PASS only when all checks are passing and none are pending or failing.\n- Return PENDING when checks are still running or queued after the wait step.\n- Return FAIL when any checks are failing, cancelled, or otherwise terminally unsuccessful.\n- Return BLOCKED when the actor output says the state could not be inspected reliably or proceeded with safely.\n\nDo not infer a different classification from narrative prose when the handoff data is explicit."}
          :on {"PASS"    {:goto :done}
               "PENDING" {:goto "wait" :max-iterations 24}
               "FAIL"    {:goto "repair" :max-iterations 8}
               "BLOCKED" {:goto :done}}}
         {:name "repair"
          :workflow "builder"
          :session {:input {:from {:step "inspect" :kind :accepted-result}}
                    :reference {:from :workflow-original}
                    :skills ["work-independently" "clojure-coding-standards" "testing-without-mocks" "code-shaper"]}
          :prompt "Address the PR check failures described by $INPUT. Work independently. Use the `clojure-coding-standards`, `testing-without-mocks`, and `code-shaper` skills.\n\nGoal: take the PR from failing toward green by either fixing the underlying issue and pushing, or rerunning checks when there is strong evidence of flake or infrastructure-only failure.\n\nRequired procedure:\n1. Read the upstream handoff to identify the PR number, PR URL, branch, worktree path, head commit, failing checks, pending checks, rerun candidates, and any candidate run ids.\n2. In the worktree, inspect the failing checks in detail using GitHub CLI and any linked check information available from `gh pr checks` / `gh pr view`.\n3. Make rerun target discovery explicit rather than implicit:\n   - begin from `candidate_run_ids` when the inspect step already found plausible run ids\n   - otherwise list recent branch runs with `gh run list --branch <pr-branch>`\n   - match failing checks to runs by workflow name, recency, and linked check context\n   - confirm the candidate run with `gh run view <run-id> --json jobs` before rerunning when job-level precision matters\n   - if rerunning a specific job, use the job `databaseId` from `gh run view <run-id> --json jobs`; do not use the browser URL job number\n4. When a rerun may be appropriate, prefer canonical GitHub Actions commands over ad hoc guesses:\n   - rerun an entire run with `gh run rerun <run-id>`\n   - rerun only failed jobs with `gh run rerun <run-id> --failed` when that matches the failure shape\n   - rerun a specific job with `gh run rerun <run-id> --job <databaseId>` only after confirming the correct job id\n5. Prefer understanding and reproducing the failure locally before changing code. Run the narrowest local verification that corresponds to the failing check, then broaden as needed.\n6. If the evidence indicates a real code or test defect, make the minimal fix, run relevant local verification, commit the change if there is one, and push the PR branch. Use the named skills to keep the change simple, robust, and mock-light.\n7. If the evidence strongly indicates flake or infrastructure noise rather than a code defect, you may rerun the affected check or checks instead of changing code. Keep reruns bounded and justified; do not loop mindlessly.\n8. If both a code fix and a rerun are needed, do the code fix first, then push so CI reruns from the new commit.\n9. If you cannot make safe progress, report that clearly rather than making speculative changes.\n10. Keep the branch/worktree state coherent. Do not leave unstaged exploratory debris behind.\n\nOutput requirements:\n- Output a compact Markdown summary with these headings exactly:\n  - `## Repair Outcome`\n  - `## Local Verification`\n  - `## CI Actions`\n  - `## Handoff Data`\n- Under `## Handoff Data`, include machine-friendly bullet lines for:\n  - `pr_number:`\n  - `pr_url:`\n  - `pr_branch:`\n  - `worktree_path:`\n  - `head_commit_before:`\n  - `head_commit_after:`\n  - `repair_mode:`\n  - `push_status:`\n  - `rerun_status:`\n  - `rerun_targets:`\n  - `remaining_risk:`\n\nSet `repair_mode:` to one of `code-fix-and-push`, `rerun-only`, `code-fix-and-rerun`, or `blocked`. Use `rerun_targets:` to record the run ids or job ids that were rerun when applicable. After this step, the next action should be to wait for checks again."
          :judge {:system-prompt "You are a workflow routing judge. Respond with exactly one word: WAIT or BLOCKED."
                  :prompt "Respond exactly with one word: WAIT or BLOCKED.\n\nRead the actor output and route strictly from the `repair_mode` in its `## Handoff Data`.\n- Return BLOCKED only when `repair_mode` is `blocked` or the actor explicitly reports it could not make safe progress.\n- Otherwise return WAIT so the workflow waits for the new or rerun checks to complete.\n\nDo not infer a different classification from narrative prose when the handoff data is explicit."}
          :on {"WAIT"    {:goto "wait" :max-iterations 24}
               "BLOCKED" {:goto :done}}}]}

Prepare a PR worktree, inspect and wait on CI checks, repair failures by pushing fixes or rerunning flaky checks when justified, and iterate until all checks pass or the workflow reaches a clearly reported blocked state.
