- ⚒ **Dispatch architecture convergence advanced materially toward the reference architecture** (2026-03-22, commits `fd368db`, `4b62b54`, `0e2a637`, `8cb4b2f`, `e5a3945`, `535d282`, `205b5be`, `e7f1917`): the `agent-session` dispatch pipeline now includes explicit permission, log, statechart, handler, effect, replay-trim, validation, and apply stages, with the effective pure-result sequencing documented and tested as `apply -> validate -> trim-effects-on-replay -> effects`. The pipeline now supports both migration-shaped pure results (`:session-update`, `:root-state-update`) and canonical reference-shaped pure results (`{:db ... :effects ...}`) through a root replacement hook, and one cohesive real production statechart-action slice now uses canonical `:db` results instead of updater-form results. Statechart participation is now an explicit dispatch boundary instead of an adjacent implicit mechanism: a ctx-provided `:dispatch-statechart-event-fn` adapter backs a first-pass `statechart-interceptor`, and core session lifecycle events (`:session/prompt`, `:session/abort`, `:session/compact-start`, `:session/compact-done`) now route through dispatch rather than direct local chart sends at the main call sites. Extension-origin permission enforcement is also now sharper: dispatch still allows compatibility fallback when no manifest event rights are declared, but when an extension manifest declares `:allowed-events`, the permission interceptor now enforces membership instead of checking registry presence alone. Internally, dispatch now carries one canonical normalized event value (`:event`) while preserving the existing public `(event-type, event-data, opts)` API and compatibility projections; logging, permission, statechart, handler, and validation paths now consume that normalized event shape coherently. The retained dispatch event log is also materially richer and closer to the reference architecture’s debugging utility: entries now include replay flag, replay class (`:replayable`, `:projection`, `:diagnostic`), statechart-claimed flag, validation error, pure-result kind, declared/applied effects, and bounded before/after root-state summaries, all exposed through the EQL dispatch-log surface together with an explicit top-level replayable-entry count. Replay classification is no longer only a static event-type table: handler registration now declares replay class directly for the meaningful replay/projection families, the older event-type mapping has been reduced to empty compatibility scaffolding, and effect-only runtime/orchestration handlers plus adjacent non-replayable session mutations are now explicitly marked `:diagnostic` rather than inheriting that status implicitly by omission. That means the retained log is no longer “all diagnostic” nor pretending to be “all replay-safe”: one minimum replay contract is now explicit and tested. Current retained replay uses only `:replayable` entries, replay preserves state application, and replay suppresses effects; `:projection` and `:diagnostic` entries remain retained observability history without a replay guarantee. The replayable subset is now broader than the initial single-name slice: a small session-config family is explicitly admitted too (`set-worktree-path`, `set-cache-breakpoints`, `set-auto-retry`, `set-auto-compaction`, `set-ui-type`, alongside `set-session-name`). Query consumers no longer need to reconstruct that subset themselves from the full dispatch log: the EQL surface now exposes replayable retained entries directly as a first-class read model in addition to the replayable count. The next step from “queryable replay subset” to “actual replay substrate” is now also present in code, not just concept: dispatch exposes explicit replay helpers for replayable retained entries, and session core can replay the retained replayable dispatch log back into canonical session state with effects suppressed. Retention semantics now align more directly with that intent too: replayable retained entries live in their own bounded retained store instead of competing with `:projection` / `:diagnostic` entries inside the mixed observability log. Replay policy is now inspectable before any events fire: the live handler registry exposes registered dispatch events, replay classes, grouping by replay class, and per-class counts through the query surface. Conforming vertical slices are now starting to emerge on top of that substrate: manual compaction is now dispatch-visible end-to-end enough to serve as the first proving slice, and the prompt/turn lifecycle has advanced another step so prompt entry now reads as `:session/prompt-submit -> :session/prompt -> :session/prompt-execute`.
